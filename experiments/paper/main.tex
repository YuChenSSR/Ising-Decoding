\title{Characterizing GPU-Accelerated Neural Pre-Decoders for Real-Time Surface Code Error Correction}

\author{Chengyu~Zhang%
\thanks{Chengyu Zhang is with Northbridge Institute of Technology, Seattle, WA 98109 USA (e-mail: chengyu.zhang@example.com).}}

\maketitle

\begin{abstract}
Quantum error correction decoders must finish within extremely small control-loop budgets, yet the practical deployment cost of neural pre-decoders remains unclear. This letter characterizes a GPU-resident pre-decoder pipeline for surface codes on an NVIDIA RTX 4090 across inference backends, numeric precision, and code distance. TensorRT lowers network execution time by about 2.7x relative to PyTorch while preserving logical error rate. Eight-bit deployment keeps accuracy essentially unchanged and reduces the model footprint, making it the most practical inference format. Across code distances 5 through 21, the pre-decoder removes 84 to 98 percent of syndrome bits and accelerates the downstream matching decoder by 1.5x to 2.1x. The full pipeline meets a one-microsecond-per-round budget through distance 7, after which the bottleneck shifts decisively to global decoding rather than neural inference.
\end{abstract}

\begin{IEEEkeywords}
quantum error correction, surface code, neural decoder, GPU acceleration, TensorRT, real-time decoding
\end{IEEEkeywords}

\section{Introduction}

Fault-tolerant quantum computing requires syndrome decoding every round, often on the order of a microsecond for superconducting qubits~\cite{google2023suppressing,battistel2023real}. Quantum error correction can tolerate high physical error rates only if the classical decoding loop keeps pace with repeated stabilizer measurements~\cite{terhal2015quantum}. Minimum-weight perfect matching (MWPM) decoders such as PyMatching remain the accuracy baseline, but their latency grows quickly with code distance~\cite{higgott2023sparse}. A neural pre-decoder can reduce syndrome density before invoking MWPM, potentially shifting work toward throughput-friendly GPU inference~\cite{nvidia2026predecoder}. The open systems question is not whether such models can decode, but which deployment stack makes them practical under real-time constraints.

This letter studies that question for two 3D CNN pre-decoders on an RTX~4090 GPU. We compare PyTorch and TensorRT backends, evaluate FP32/FP16/INT8 deployment, and trace latency scaling from distance 5 to 21. Prior work established neural decoders and hybrid local-global decoding strategies~\cite{torlai2017neural,chamberland2023techniques}; our contribution is an architecture-focused characterization that identifies when neural inference helps and where the end-to-end bottleneck moves. Across all experiments, TensorRT preserves decoding quality, INT8 is the best deployment format, and the limiting factor beyond distance 7 is no longer the neural network but the downstream global decoder.

\section{Background and System Architecture}

The rotated surface code encodes one logical qubit into a distance-dependent 2D lattice and produces a 3D syndrome volume across repeated measurement rounds~\cite{fowler2012surface}. Under circuit-level noise, measurement faults create time-like correlations in that volume, making decoding substantially harder than in code-capacity settings~\cite{dennis2002topological}. Our pipeline, shown in Fig.~\ref{fig:pipeline}, applies a 3D CNN to the raw syndrome, computes a residual syndrome from the predicted corrections, and sends that residual to PyMatching. This design keeps the learned model focused on local structure while retaining a conventional global decoder for hard cases.

We evaluate two fully convolutional 3D CNNs from the Ising-Decoding framework~\cite{nvidia2026predecoder}. Model~1 uses four layers and about 912K parameters with receptive field 9; Model~4 uses six layers and about 1.8M parameters with receptive field 13. Both consume four-channel syndrome tensors, use same-padding 3D convolutions, and operate on local spatiotemporal neighborhoods rather than on handcrafted decoder features. The network predicts local corrections, from which a residual syndrome is recomputed before invoking the global decoder; the systems objective is therefore to reduce the expensive part of the pipeline rather than to replace it outright. Deployment follows a realistic serving path: PyTorch eager mode and \texttt{torch.compile} represent software baselines, while TensorRT FP16 and TensorRT INT8 represent compiled inference engines built through ONNX export~\cite{tensorrt}.

\section{Experimental Setup}

Both models are evaluated under a 25-parameter circuit-level noise model at physical error rate 0.003, with logical error rate measured over 262,144 Monte Carlo shots per basis~\cite{nvidia2026predecoder}. We report logical error rate, PyMatching single-shot latency, network forward time, and syndrome density reduction. The study covers four experiment groups: backend comparison at distance 7 with Model~1; quantization and model-capacity comparison at distance 7; latency scaling from distance 5 to 21 with TensorRT FP16; and direct FP16 versus INT8 real-time feasibility analysis from distance 5 to 13.

All experiments run on an NVIDIA RTX~4090 GPU. PyMatching latency is measured after warmup using batch size 1 decoding so that the reported microsecond-per-round numbers correspond to online decoder behavior rather than throughput-optimized batching. Network forward time is reported separately over the full evaluation set to show where GPU throughput helps and where the downstream decoder remains fundamentally latency-bound. Training follows the framework's noise-upscaling procedure so that learning sees denser syndromes, while evaluation always uses the original unscaled noise model.

\section{Results}

\subsection{Backend Comparison and Quantization}

Table~\ref{tab:backend} shows that all four backends deliver nearly identical decoding quality at distance 7. Logical error rate stays near $2.3\times10^{-3}$ and PyMatching speedup stays near $2\times$, indicating that backend choice does not materially change correction quality. TensorRT reduces model forward time from 0.451~s in PyTorch eager mode to 0.168~s in FP16 and 0.172~s in INT8, a 2.7x improvement. The \texttt{torch.compile} result is dominated by one-time compilation cost, so it is less attractive for latency-sensitive deployment.

Figure~\ref{fig:backend} makes the backend result visually clear, and Fig.~\ref{fig:quant} extends the comparison to model size and numeric precision. Model~4 lowers logical error rate by about 36\% relative to Model~1 at similar PyMatching latency, while INT8 remains effectively indistinguishable from FP32 and FP16 for both models. For example, Model~4 stays near $1.4\times10^{-3}$ across all three precisions, and Model~1 changes only from about $2.18\times10^{-3}$ to $2.28\times10^{-3}$ under INT8. This makes INT8 the preferred deployment point: it preserves decoding quality, minimizes model footprint, and gives the same practical latency as FP16 on the target GPU.

An additional practical takeaway is that deployment optimization helps the neural stage far more than the decoder stage. The backend choice changes model execution time substantially, yet the per-round matching latency barely moves because PyMatching sees almost the same residual workload under all four inference stacks. Backend engineering therefore shrinks front-end cost aggressively without changing the downstream decoding behavior.

\subsection{Scaling and Real-Time Feasibility}

Figure~\ref{fig:scaling} shows that the pre-decoder consistently reduces PyMatching latency from distance 5 through 21. Syndrome density reduction remains between 84\% and 98\% throughout this range, and PyMatching speedup stays between 1.5x and 2.1x. Representative points illustrate the trend: latency drops from 1.18 to \SI{0.77}{\micro\second}/round at distance 5, from 2.15 to \SI{1.07}{\micro\second}/round at distance 9, and from 12.26 to \SI{7.11}{\micro\second}/round at distance 21. The pre-decoder therefore continues to help at larger distances, but the absolute cost of global decoding rises much faster than network inference.

This trend is notable because the deployed convolutional models were trained around distance 7 yet still transfer to significantly larger lattices. The likely reason is architectural rather than statistical: local convolutional filters continue to capture repeatable short-range error motifs even as the global decoding graph expands. What degrades with distance is not the usefulness of the neural front end, but the fraction of the full syndrome volume that can be resolved before a global matching step becomes necessary.

That bottleneck transition becomes explicit in Fig.~\ref{fig:realtime}. With batch-amortized inference, neural forward time contributes less than \SI{0.1}{\micro\second}/round, so total latency is governed by PyMatching once the residual syndrome is produced. The complete pipeline satisfies the one-microsecond budget through distance 7, reaches \SI{1.07}{\micro\second}/round at distance 9, and diverges further beyond that point. FP16 and INT8 trace nearly identical curves, confirming that deployment precision is no longer the dominant systems concern after the model is quantized and compiled. The architecture implication is clear: future gains depend more on GPU-friendly global decoding than on further optimizing the neural front end, consistent with emerging GPU-native decoders such as cudaq-qec~\cite{cudaqqec}.

This phase breakdown also explains why the pre-decoder remains useful even when the absolute latency target is missed. The neural stage removes a large fraction of easy local defects at effectively amortized cost, but the residual errors still force PyMatching to execute a largely sequential global procedure. In other words, the pre-decoder shifts the frontier, yet it cannot by itself eliminate the algorithmic mismatch between massively parallel GPU inference and latency-sensitive matching on sparse residual graphs.

\section{Implications and Limitations}

For near-term deployment, the main architectural lesson is that a neural pre-decoder should be treated as a front-end filter rather than as a standalone replacement for global decoding. Once TensorRT compilation and INT8 quantization are applied, the learned model is already cheap enough that further optimizing it yields diminishing returns unless the downstream decoder also changes. The measured bottleneck therefore points toward GPU-resident matching, Union-Find, or belief-propagation-style decoders as the next co-design target.

Model capacity introduces a second systems trade-off. Model~4 consistently provides lower logical error rate than Model~1, and at distance 7 that extra capacity does not meaningfully harm real-time feasibility. For a single logical qubit, Model~4 is therefore the better quality point. Model~1 remains useful when deployment is constrained by accelerator memory, multi-instance scheduling, or controller simplicity, because its smaller footprint makes replication easier even if its residual syndrome is somewhat harder for the global decoder to finish.

The control path also matters. Our measurements isolate PyMatching latency from the batched neural stage, but a production stack must still move syndrome data, schedule kernels, and return corrections on time. The lesson is not that batching solves the entire problem; it is that GPU throughput can already make the local learned stage cheap enough that system designers should spend more effort on the residual path, the host-device boundary, and the structure of the global decoder itself.

The study also has clear limits. All measurements were taken on one GPU class and one circuit-level noise model, so the exact crossover point could shift on other hardware or under different noise regimes. In addition, the reported neural cost benefits from batch amortization, which assumes a control stack capable of feeding the accelerator efficiently. These caveats affect the precise numbers, but they do not change the central systems result that the global decoder dominates after distance 7.

\section{Conclusion}

This study shows that neural pre-decoders are already practical from a deployment standpoint: TensorRT provides a robust 2.7x reduction in network execution time, INT8 keeps accuracy intact while minimizing footprint, and the pre-decoder reliably lowers PyMatching work across all tested distances. The real-time boundary on this platform is not neural inference but the downstream global decoder, which becomes dominant immediately after distance 7.

Our conclusions are limited to one GPU class and one circuit-level noise model, and the batch-amortized inference cost assumes a control stack that can exploit GPU throughput. Even with those limits, the architectural direction is stable: a neural pre-decoder is not the long pole. A fully GPU-resident global decoder is the next requirement for real-time fault-tolerant operation at larger code distances, and future decoder co-design should treat neural inference, graph construction, and global matching as one integrated architecture problem rather than three isolated optimizations.
